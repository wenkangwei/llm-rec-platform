## 0. 前置说明
- 对所有任务，你都应该先规划拆解问题成子问题子模块，然后逐步解决
- 每解决完一个子问题子模块必须验证测试
- 注意子模块拆分的细致力度，拆太细校验会很耗时，拆太粗难稳定实现功能容易返工


## 1. 架构设计原则

### 1.1 模块化与解耦
- 每个模块（召回、排序、特征、监控）必须独立，业务逻辑与框架代码分离
- 模块间通过 `RecContext` 上下文对象传递数据，不直接引用其他模块内部实现
- 新增召回通道/排序模型/特征源时，不影响已有模块

### 1.2 统一抽象接口
- 所有链路模块（召回、粗排、精排、重排、混排）必须实现 `PipelineStage` ABC
- 接口签名统一：`process(ctx: RecContext) -> RecContext`
- 模块注册制：通过配置文件声明模块，链路执行器动态加载

### 1.3 配置驱动
- 单一配置入口 `configs/app.yaml`，图依赖结构
- 配置文件之间通过 `${path:key}` 引用，通过 `${env:VAR:default}` 支持环境覆盖
- 加载顺序：拓扑排序 → 解析引用 → 环境覆盖 → Schema 校验

### 1.4 可拆分部署
- 当前单机部署，但模块间协议统一（Protobuf 定义 + Pydantic Schema）
- 未来可按模块拆分到不同机器，无需修改业务代码
- gRPC 服务定义预留，HTTP 优先实现

## 2. 推荐链路规范

### 2.1 四阶段漏斗
```
召回(Recall) → 粗排(Prerank) → 精排(Rank) → 重排(Rerank) → 混排(Mixer)
```

| 层级 | 职责 | 时延要求 | 候选集规模 | 规范要点 |
|------|------|---------|-----------|---------|
| **召回层** | 从海量物料中快速筛选潜在候选 | < 50ms | 千万→万级 | 多路并行、索引优化、去重兜底 |
| **粗排层** | 快速打分筛选，为精排减负 | < 10ms | 万→千级 | 轻量模型、特征裁剪、批量计算 |
| **精排层** | 精准预估 CTR/CVR 等目标 | < 50ms | 千→百级 | 复杂模型、实时特征、A/B 测试支持 |
| **重排层** | 业务规则、多样性、fatigue 控制 | < 10ms | 百→十级 | 规则可配置、探索与利用平衡 |
| **混排层** | 多类型内容混合输出 | < 10ms | 十级 | 场景适配、内容类型比例控制 |

层与层之间通过**标准化协议**（Protobuf/JSON）通信，支持独立迭代和灰度发布。

### 2.2 召回通道规范
- 每个召回通道独立实现 `PipelineStage` 接口
- 召回结果写入 `ctx.candidates`，附带 `source` 标记来源通道
- Merger 负责多通道结果去重合并
- 召回通道权重通过配置文件管理，支持 A/B 实验

### 2.3 上下文透传
- `RecContext` 贯穿整个链路，禁止跨模块直接传参
- 每个模块只读写 `ctx` 中属于自己的字段
- 模块间通过 `ctx.extras` 传递扩展信息，key 需以模块名前缀命名

## 3. 数据流规范

### 3.1 数据流向
```
用户请求 → RecContext 构建
  → 特征拉取（FeaturePlatform）
  → 链路执行（Recall → Prerank → Rank → Rerank → Mixer）
  → 日志追踪（RecTracer）
  → 响应返回 + 异步日志落盘

离线流程：日志采集 → 数据仓库 → 特征工程 → 模型训练 → 模型仓库 → 推送到线上
实时流程：行为日志 → Flink 实时特征 → 在线特征库 → 实时更新模型/缓存
```

### 3.2 批流一体架构（Lambda/Kappa）
- **批处理层**：T+1 离线训练全量模型，处理历史数据，保证数据完备性
- **流处理层**：实时特征更新（Flink/Spark Streaming），分钟级模型增量更新
- **服务层**：统一特征存储（Feature Store），离线在线特征一致性校验

### 3.3 特征获取
- 禁止业务代码直接访问 Redis/MySQL 等存储获取特征
- 统一通过 `FeaturePlatform.get_features()` 获取
- 特征定义在 `FeatureRegistry` 中注册，包含血缘关系
- 衍生特征通过 DSL 引擎实时计算

- 衍生特征通过 DSL 引擎实时计算

### 3.4 特征工程规范
- **特征分类**：用户画像、物料画像、上下文特征、交叉特征
- **存储规范**：冷热分离，高频特征内存存储（Redis），低频特征 KV 存储
- **一致性要求**：离线训练特征与在线服务特征**必须同源**（同一份 ETL 代码），防止线上线下特征偏差

### 3.5 日志落盘
- 每个请求生成完整的 `PipelineTrace`（链路追踪 + 物品追踪）
- 训练日志（`TrainingLogEntry`）异步写入 Parquet 文件
- 用户行为标签延迟回填，T+1 合并生成训练样本

## 4. 模型服务规范

### 4.1 模型接口
- 所有模型必须实现 `ModelService` ABC
- 通过 `ModelManager` 统一管理模型生命周期（加载、预热、热更新、卸载）
- 模型推理统一走 `predict()` 接口，后端可切换（PyTorch/ONNX/Triton）
- 模型格式支持多种（ONNX/TensorRT/SavedModel），便于异构硬件（GPU/CPU）部署
- 精排层支持批量推理（Batching），提升 GPU 利用率

### 4.2 模型热更新
- 新模型版本发布不中断服务
- 灰度切换：先加载新模型到内存，验证通过后切换流量
- 旧模型延迟卸载，回滚时可立即切回

### 4.3 双塔模型
- UserTower 和 ItemTower 独立编码
- 用户 embedding 周期性更新（近线）
- 物品 embedding 新内容实时计算
- embedding 存储到 Faiss，ANN 检索

## 5. LLM 融合规范

### 5.1 LLM 后端
- 统一 `LLMBackend` 抽象，支持 vLLM/Triton/Mock 切换
- LLM 调用统一通过 `LLMFactory` 创建后端实例
- 异步调用 + 超时控制 + 降级策略

### 5.2 Agent 框架
- Agent/Tool 遵循统一抽象接口
- Planner → Executor → Critic 模式
- Monitor Agent 定期分析监控数据，生成告警

### 5.3 Prompt 管理
- Prompt 模板外置到 `llm/prompt/templates/` 目录
- 支持变量替换和版本管理
- 禁止在代码中硬编码 Prompt 文本

## 6. 监控规范

### 6.1 链路追踪
- 每个请求必须有完整的 `PipelineTrace`
- 记录每个阶段的延迟、输入输出数量
- 追踪每个候选物品在各阶段的分数和位置变化

### 6.2 在线监控
- **延迟监控**：P99 分位延迟告警（召回 < 100ms，全链路 < 200ms）
- **成功率**：服务可用性 ≥ 99.99%，异常自动熔断与恢复
- **业务指标**：CTR、转化率、停留时长等实时看板
- Prometheus 格式指标：QPS、延迟分布、召回覆盖率、模型推理耗时

### 6.3 离线评估
- **评估集构建**：时间窗口切分（避免数据泄露），正负样本均衡
- **指标矩阵**：AUC、GAUC、NDCG、多样性指标、覆盖率、新颖性

### 6.4 召回覆盖分析
- 记录每个召回通道的召回数量、存活数量、最终曝光数量
- 分析通道贡献度和重叠率

## 7. 编码规范

### 7.1 Python 风格
- Python 3.10+，使用 type hints
- 数据类优先使用 `@dataclass`，API 模型使用 Pydantic `BaseModel`
- 异步优先：IO 操作使用 `async/await`
- 依赖注入：通过构造函数注入，禁止模块内部实例化依赖
- 模块测试: 每开发完一个完整模块的接口和实现先进行模块测试debug，再继续后面开发

### 7.2 目录约定
- 每个模块目录包含 `base.py`（抽象接口）和具体实现
- 测试目录结构与源码目录结构对齐
- 配置文件与代码分离，放在 `configs/` 目录

### 7.3 错误处理
- 模块内部异常不向上抛出，记录日志后降级处理
- 链路执行中单个模块失败不影响整体请求
- 关键路径有降级策略（如模型推理失败走默认排序）

## 8. A/B 测试规范

- **分流层**：在召回前进行流量正交切分，支持多层实验（互不影响）
- 实验配置通过配置中心管理
- 流量分桶基于用户 ID hash
- **指标埋点**：每个推荐结果需携带实验 ID、模型版本、特征版本等溯源信息
- 实验指标：CTR、CVR、留存、多样性
- 实验报告自动生成，包含统计显著性检验

## 9. 高可用规范

- 无状态服务设计，支持水平扩展（Kubernetes HPA）
- **多级缓存**：本地缓存（Caffeine）+ 分布式缓存（Redis），降低数据库压力
- 缓存降级：Redis 不可用时回源 DB，DB 不可用时返回缓存/默认值
- 模型降级：精排模型不可用时降级为粗排分数，模型超时自动切换至规则策略
- 召回降级：某通道不可用时跳过，不阻塞整体链路
- 熔断机制：基于 QPS 的限流，下游服务故障时自动熔断，防止雪崩
- **数据兜底**：冷启动用户、新物料的默认策略，确保服务不空返回

## 10. 合规与安全

- 用户数据加密存储，PII 字段脱敏
- 推荐结果可解释：记录每个物品的推荐理由（特征重要性、规则命中说明、来源通道）
- 日志脱敏：训练日志中用户 ID 不可逆加密
- 内容安全：LLM 生成内容需经过安全过滤
- **公平性**：避免算法偏见，对不同人群、内容生产者保持机会公平
- **隐私计算**：用户敏感特征脱敏，联邦学习场景下数据不出域
