# LLM 推荐系统平台 — 产品需求文档

## 1. 项目概述

构建一个融合 LLM 能力的内容推荐系统平台，面向 10万-100万 用户规模，支持内容推荐 + 社交属性（关注/粉丝、互动、社区、话题）的个性化推荐场景。

## 2. 业务场景

### 2.1 核心场景
- **首页信息流推荐**：个性化内容 Feed，混合图文、视频等多种内容类型
- **搜索推荐**：基于用户 query 的搜索结果个性化排序
- **关注动态流**：已关注用户/社区的内容聚合
- **社区/话题推荐**：基于兴趣和社交关系的社区与话题推荐

### 2.2 社交特征
| 特征类别 | 具体内容 |
|---------|---------|
| 关注关系 | 关注列表、粉丝列表、互关关系 |
| 互动行为 | 点赞、评论、分享、收藏 |
| 社区归属 | 加入的社区、社区活跃度 |
| 话题兴趣 | 关注/参与的话题、话题偏好 |

## 3. LLM 融合能力

### 3.1 Item Embedding 生成
- 利用 LLM 理解内容语义，生成高质量 item embedding
- 替代/补充传统 TF-IDF、Word2Vec 等浅层文本表征
- 支持多模态内容（文本+图片描述）的统一向量化

### 3.2 内容模拟生成
- 新内容冷启动：利用 LLM 基于内容特征生成模拟交互数据
- 辅助冷启动 item 的初始排序分数评估

### 3.3 搜索重排摘要
- 搜索场景下，对候选内容生成个性化摘要
- 基于用户兴趣点重新组织搜索结果的展示文本
- 提升 CTR 和用户满意度

## 4. 推荐链路架构

### 4.1 链路流程
```
多路召回(Multi-channel Recall)
  → 融合(Merger)
  → 粗排(Prerank)
  → 精排(Rank)
  → 策略重排(Strategy Rerank)
  → 多路混排(Mixer)
```

### 4.2 召回通道
| 通道 | 说明 | 技术方案 |
|------|------|---------|
| 个性化召回 | 基于双塔模型的向量召回 | Two-Tower + Faiss ANN |
| 协同过滤召回 | 基于行为相似度 | ItemCF / UserCF |
| 社交召回 | 基于关注关系 | 关注者最近交互内容 |
| 社区召回 | 基于社区归属 | 活跃社区热门内容 |
| 热门召回 | 全局/分类热门 | 实时热度排行榜 |
| 运营召回 | 运营精选/置顶 | 配置化投放 |
| 冷启动召回 | 新用户/新内容 | LLM 内容模拟 + 探索策略 |

### 4.3 排序层级
- **粗排 (Prerank)**：LightGBM，快速过滤，从数千候选降至百级别
- **精排 (Rank)**：DNN 模型（DCN-v2 / DIN / DeepFM），精准预估 CTR/CVR
- **策略重排 (Rerank)**：业务规则（多样性、去重、打散、流量控制）
- **混排 (Mixer)**：多场景/多类型内容混合输出

## 5. 非功能需求

### 5.1 性能目标
| 指标 | 目标值 |
|------|--------|
| 推荐延迟 (P99) | < 200ms |
| 并发 QPS | ≥ 1000 |
| 召回覆盖率 | ≥ 85% |
| 日志落盘完整性 | ≥ 99.9% |

### 5.2 可用性
- 服务可用性 ≥ 99.9%
- 模型热更新不中断服务
- 降级策略：召回/排序模块可独立降级

### 5.3 可扩展性
- 模块间协议统一，支持独立拆分部署
- 新召回通道/排序模型可插件化接入
- 特征平台支持动态扩展特征源

## 6. 部署规划

### 6.1 初始部署
- 单台 ECS + Docker Compose（阿里云 gn7i V100）
- 预算：约 3000 元/月
- 包含：推荐服务 + vLLM 推理 + 数据存储 + 监控

### 6.2 技术栈
| 类别 | 技术 |
|------|------|
| 后端框架 | Python FastAPI |
| LLM 推理 | vLLM |
| 深度学习 | PyTorch |
| 树模型 | LightGBM |
| 向量检索 | Faiss |
| 缓存 | Redis |
| OLAP | ClickHouse |
| 关系数据库 | MySQL |
| 监控 | Grafana + Prometheus |
| 日志 | Vector |
| 容器化 | Docker Compose |
